TCP
  • [TCP]：http, ftp, ssh 是基于 TCP 实现的，TCP 是可靠的协议，数据包稳定有序的传达，如果包发生丢失或者超时就会触发重发机制。
  • [UDP]: dns 是基于 udp 实现的。UDP 是不可靠协议，发出去数据包之后就不管结果，常用的应用场景 视频聊天 游戏。

思考：如果 TCP 包过大如何去优化？

优化方式：

  1. 浏览器缓存策略

  • 浏览器缓存包括：HTTP 缓存（强缓存 协商缓存）、indexDB、cookie、localstorage

    强缓存和协商缓存。强缓存如果命中缓存不需要和服务器端发生交互，而协商缓存不管是否命中都要和服务器端发生交互，强制缓存的优先级高于协商缓存。（一般应用广泛的是用 HTTP 头信息控制缓存）

    【强缓存】

  • Expires 缓存过期的时间，超过了这个时间点就代表资源过期。

  • Cache-Control ：max-age 指定一个时间长度，在这个时间段内缓存是有效的，
    【协商缓存】
  • Last-modified: 服务器端资源的最后修改时间，响应头部会带上这个标识。
  • Etag/If-None-Match：由服务器端上生成的一段 hash 字符串，第一次请求时响应头带上 ETag: abcd，之后的请求中带上 If-None-Match: abcd，服务器检查 ETag，返回 304 或 200。

应用场景的区分

不经常改变的文件:
给 max-age 设置一个较大的值，一般设置 max-age=31536000

可能经常需要变动的文件:
Cache-Control: no-cache / max-age=0 2. http2 多路复用

优化分为两个方面：

  1. 如何让文件加载的更少
  • 缓存 cdn, 框架基本都有 contentHash（md5 去计算内容是否有改变来缓存）的缓存机制。

  • 图片优化

    0）图片压缩 可以用七牛云 或 阿里云，现在的 cdn 大部分都可以在图片链接添加后缀来处理图片，比如裁剪、压缩等。

    1）懒加载

    2）渐进加载：miniwebp 先加载低像素的图片做占位符。缺点是会消耗更多的性能，but 可以提高用户体验。

  1. 计算 FMP（第一次有效渲染）的几个方案

思考：怎样算有效渲染？针对不同的网站需求，FMP 的开始计算节点不同

场景：1.首屏是大视频 video 2.首屏主要内容是轮播图（比如淘宝）3.首屏主要内容是搜索框（比如百度）。总结看情况而定，找权重最重的模块

tips: 静态文件比如 video,img 不用等页面加载完毕再计算，performance.getEntries()会返回所有静态资源的信息。

FMP 开始计算的几种解决方案

  • 固定好 dom 的 id,计算该 dom 高度变化的时间（mutationObserver 可以监听 DOM 的变化）
  • 统计首屏 dom 的面积，计算占比面积
  • 使用第三方工具 lighthouse(生成页面质量报告) 使用谷歌自带的页面检测
  • 普通的开始统计节点，页面 TIT 之后再开始统计 FMP 耗时
