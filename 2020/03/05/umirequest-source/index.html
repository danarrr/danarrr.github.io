请求库源码解读

发表于 2020-03-05 | 分类于 请求库 | 阅读次数:

前言

最近新项目使用到了(基于 fetch 封装)请求库 umirequest, 官网一张对比图眼前一亮。
附上之前写的 ajax axios 对比传送门。

先来看看它的优点：

  1. 适用于各大业务场景,丰富请求能力。
  2. 提供了请求层的常用功能
  • URL 参数自动序列化
  • POST 数据提交方式简化
  • Response 返回处理简化
  • 请求超时处理
  • 请求缓存支持
  • GBK 编码处理
  • 统一的错误处理方式
  • 请求取消支持 （以前选用 axios 的最大优点，这里完美覆盖
  • Node 环境 http 请求
  • 拦截器机制
  • 洋葱中间件机制
  1. 面向多端、多设备应用，umi-request 不仅支持 browser http 请求，也同时满足 node 环境、自定义内核请求等能力。

引发思考

这么多网络请求库究竟啥时候学的完？

设计理念

不如来看看他的设计模型，类似于 koa 的洋葱模型的中间件机制:
每一层洋葱圈为一个中间件，请求经过一个中间件都会执行两次，开发者可以根据业务需求很方便地实现请求前后增强处理。

阅读源码

从三个最常用的功能作为切入点：

  • 拦截器是如何运作的?
  • 洋葱模型设计理念用来做了啥?
  • 中间件做了啥?
  • 如何中断请求（调研过后发现是基于 cancelable promises 的提案实现的，但该提案暂时被撤回这里不展述了 0.0）

接下来说说洋葱设计理念

参考 koa 洋葱的设计理念的讲解。
看完会有了个模糊的概念:

  1. 洋葱模型类似于的说是代码（函数）的执行顺序
  2. 中间件 可以用来组合这些函数
  3. umirequest 源码中有个函数 compose()（官方解释：用来返回一个所有插件的插件）, 其实就是 umirequest 的中间件层 Middlewares。

总结

第一次看源码一头雾水 借鉴导师的一段话：
“看源码的过程中，很忌讳一个函数一个函数死扣着一些变量和方法不放。因为这样很难跳出那些方法，容易看着看着就被绕进去，忘了你要看的是什么。其实看源码很多是了解他的运行原理，有问题的时候可以快速定位，排查问题。再高一点就是抽象他的框架思想，为后面造轮子或造工作做准备。”
